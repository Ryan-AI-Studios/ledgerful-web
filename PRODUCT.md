# Product

## Register

brand

## Users

Ledgerful Web serves engineering managers, senior developers, platform leads,
and security reviewers evaluating Ledgerful before they install it across one or
many repositories. They arrive with skepticism about AI coding tools, source
upload, compliance claims, and integration sprawl. They need a public site that
quickly separates what is real today from what is beta, local-only, or planned.

## Product Purpose

Ledgerful Web is the public marketing, documentation, pricing, changelog, and
trust/security surface for Ledgerful. It exists to make the local-first engine
credible and installable without confusing the public site with the embedded
dashboard or a future hosted control plane. Success means a team can understand
the product boundary, verify release/install claims, evaluate security posture,
and choose the right adoption path without a sales call.

## Brand Personality

Precise, evidence-first, and controlled. The site should feel like a serious
engineering instrument: confident enough to make commercial claims, disciplined
enough to label limits, and clear enough for security reviewers to trust. It
should avoid hype, fake enterprise polish, and generic SaaS optimism.

## Anti-references

- Generic AI SaaS landing pages with vague productivity promises.
- Mock dashboards presented as live operational evidence.
- Enterprise-feature overclaims for SSO, RBAC, SCIM, hosted audit logs, GitHub
  Apps, or billing before the control plane exists.
- Decorative card grids, gradient text, glassmorphism, fake customer logos,
  fake uptime/status links, and unresolved package/registry links.
- Terminal cosplay that uses monospace as the entire brand instead of showing
  actual product evidence.
- Editorial-magazine styling that makes a developer tool feel like a template
  rather than an engineering product.

## Design Principles

1. **Truth before persuasion.** Every public claim must be classified as
   available, beta, local-only, hosted planned, or enterprise planned.
2. **Show evidence, not vibes.** Prefer release checks, command output,
   signed-export diagrams, screenshots, and feature matrices over abstract
   promises.
3. **Protect the local-first boundary.** The public web must never imply that
   the localhost daemon is a SaaS backend or that source upload is required.
4. **Build trust through constraints.** Known limitations, honest-zero metrics,
   launch blockers, and unresolved links should be visible instead of hidden.
5. **Keep the adoption path concrete.** Each major page should help a team
   decide the next action: install locally, review docs, verify release assets,
   assess trust posture, or track planned hosted capabilities.

## Accessibility & Inclusion

Target WCAG 2.2 AA for public pages. Use sufficient contrast for body and muted
text, visible focus states, semantic landmarks, keyboard-accessible navigation,
clear reduced-motion behavior, and multi-cue status indicators that do not rely
on color alone. Security and compliance content should be readable without
marketing animation or hover-only disclosure.
